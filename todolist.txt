[ ] If Anthropic API spend trends toward $50/mo: migrate hot paths
    Added: 2026-04-23
    Trigger: $20/mo cap already enforced in-code via api_spend_log +
    pre-flight check. This entry is the NEXT-STEP plan if the cap is
    bumping up against the ceiling week after week and quality suffers
    from the refusals.
    Options to evaluate when triggered:
      A) OpenRouter as a drop-in. Route the `claude-sonnet-4-5` calls
         through OpenRouter's Anthropic pass-through and negotiate a
         volume discount. Pros: no code change beyond the client init,
         keeps the tool-use schema identical. Cons: adds a hop of latency
         (~100-200ms), OpenRouter's uptime is slightly worse than Anthropic
         direct.
      B) Gemini 2.5 Flash for the SMS intent parser. Flash is ~10x cheaper
         than Sonnet and handles constrained tool-use fine for
         classification-grade tasks. Keep Sonnet for the agent chat where
         multi-step reasoning matters. Cons: dual-provider code path, need
         to maintain a separate tools schema format, new env vars.
      C) DeepSeek V3 (or similar) via its own API for the SMS parser.
         Cheaper still than Gemini Flash. Cons: data residency / privacy
         questions if PII ever leaks into a command body.
      D) Local Ollama (Llama 3.1 8B or Mistral) for SMS parser only.
         Zero marginal cost. Cons: requires a persistent inference host
         (not Vercel serverless), which reopens the "don't want to run it
         on my laptop" thread. Would need a cheap VPS ($6-12/mo) — at
         that point the net cost savings may wash vs. just paying Flash.
    Do not start evaluating until the $20 cap has been burning for a
    full month with legitimate traffic. Premature optimization.

[ ] Twilio delivery status callback (outbound)
    Added: 2026-04-23
    Context: Twilio's Messaging Service exposes a Delivery Status
    Callback URL. When wired, every outbound message gets a webhook
    POST as it progresses (queued → sent → delivered / failed /
    undelivered) plus the carrier error code on failure.
    Today we only know an outbound failed by checking the Twilio Console
    manually (that's how we caught the 30032 TFV error during the SMS
    test). With this webhook we'd catch it in the DB automatically.
    Scope:
      1. Build POST /api/webhooks/twilio/status with X-Twilio-Signature
         verification (reuse the helper pattern from
         app/api/webhooks/twilio/inbound/route.ts).
      2. On each callback, look up the sms_messages row by twilio_sid
         and update a new `delivery_status` + `delivery_error_code`
         + `delivery_error_message` + `delivered_at` columns
         (need a small migration).
      3. Dashboard surfaces: recent undelivered outbounds in a new
         small widget so Jason can see "X messages didn't land" at
         a glance.
      4. Configure the Callback URL in Twilio Console -> Messaging ->
         Services -> Integration -> Delivery Status Callback.
    Worth doing before public vendor-facing SMS (vendor notifications,
    reminder dispatches) so a silent undelivered never goes unnoticed.
