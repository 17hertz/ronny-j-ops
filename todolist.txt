[ ] Email ingestion via Resend inbound (forwarding-based, no Gmail OAuth)
    Added: 2026-04-24
    Why this is the right shape (not Gmail API + not Zapier):
      - Gmail OAuth with gmail.readonly is a "restricted" scope. In
        Published mode it needs CASA — an independent security audit
        costing $15K-30K/year for a small app. Not proportional to
        two users.
      - Testing mode bypasses CASA but issues 7-day-expiring refresh
        tokens. We already learned that the hard way earlier in this
        project; weekly re-auth is a dealbreaker.
      - Zapier works (their Gmail connector is pre-verified) but
        costs $240-600/yr and adds a third-party TOS in the chain.
      - Resend is free, already in the stack for outbound. Email at
        the MX level = pure SMTP, Google isn't involved on our side.
    How it works:
      - User forwards emails from Gmail to an address on a subdomain
        we control (e.g. ronny@inbox.ronnyjlistenup.com). Resend
        catches them, POSTs to a webhook we build. No OAuth, no scope
        verification.
      - User's own Gmail stays untouched. They choose what to forward.
    Prerequisites (Jason to decide):
      - Need DNS control over a domain: 17hertz.io or
        ronnyjlistenup.com. ronnyj.17hertz.com is a Vercel subdomain;
        MX records have to live on the parent apex or a dedicated
        subdomain of a domain Jason registers directly.
      - Gmail requires a one-time "verify forwarding destination" step
        where Google emails a code to the destination and the user
        pastes it into Gmail settings. Surface that code via a
        dashboard page OR a Claude SMS tool so Ronny doesn't have to
        log into Resend.
    Implementation scope:
      1. Configure Resend inbound: MX records + inbound route pointing
         to our webhook URL. 5 min DNS + 5 min Resend setup.
      2. Migration: public.inbound_emails table (id, team_member_id,
         from_address, to_address, subject, body_text, body_html,
         received_at, raw_headers jsonb, attachments_meta jsonb) +
         public.inbound_email_aliases (team_member_id -> local_part
         like "ronny" so ronny@inbox... routes to the right member).
      3. POST /api/webhooks/resend-inbound: verifies Resend signature,
         parses MIME, resolves recipient -> team_member via alias
         table, inserts.
      4. Claude tools in lib/agent/tools.ts:
         - search_inbound_emails(query, days_ago?, from?) — read
         - get_inbound_email(id) — read
      5. Update chat-passthrough system prompt so Claude knows to
         search email when the user says "that email from X about Y"
         or "add the details from the promoter's email."
      6. Dashboard page /dashboard/inbox (simple list + search) so
         Jason can eyeball what's landed.
      7. Ronny's one-time setup: in Gmail Settings -> Filters,
         create rule "From: promoter@*, venue@*, specific senders"
         -> Forward to ronny@inbox.ronnyjlistenup.com. Verify with
         the code. Iterate on filter breadth over time.
    Token cost note: each email in Claude's context is 500-5000
    tokens. Cap max-fetched-per-question at 3, use Gmail-style search
    operators in search_inbound_emails (from:, newer_than:,
    has:attachment) to narrow BEFORE fetching bodies.
    Risk: attachment handling. First pass = ignore attachments, store
    metadata only. If Ronny needs contract PDFs parsed, that's a
    follow-up (put it through a PDF text extractor, feed to Claude as
    additional context).
    Total build estimate: ~3-4 hours of code + ~30 min setup on
    Jason/Ronny's side. Don't attempt when fried.

[ ] Screenshot → OCR → Claude classify → auto-route
    Added: 2026-04-24
    Idea: Ronny snaps a pic of an email, a contract, a receipt, a flyer,
    a handwritten note. Sends to the app (via SMS MMS, WhatsApp, or a
    dashboard drop-zone). Claude looks at the image, decides what it is,
    files it correctly, and follows up.

    Classification targets + routing:
      - TASK            → createTask
      - EVENT (flyer,
        venue booking)  → createEvent
      - BILL: SERVICE   → vendor doesn't exist yet → text/email them the
                          intake signup link (reuses existing vendor invite
                          flow) so they submit their W9 + banking info.
                          Ties the invoice to the new vendor once they claim.
                          Requires W9 because SERVICES to a business are
                          1099-reportable.
      - BILL: PRODUCT   → goods purchased (gear, merch supplies, etc.).
        (has sales tax)   Sales tax on the receipt = it's a PRODUCT purchase
                          from a registered retailer, which means NO W9
                          needed (sales tax already satisfies the state's
                          revenue reporting). Store as a row in a new
                          `direct_expenses` table (date, merchant name,
                          amount, category, receipt_storage_path). Roll
                          into the existing expense report PDFs.
      - PAYMENT LINK    → Stripe / Venmo / PayPal link on a bill. Parse
                          URL, store as pending_payment row with link,
                          create a task to pay. On payment, either manual
                          confirmation or Stripe webhook (post-v1).
      - CONTACT INFO    → store in contacts / create a vendor stub.
      - OTHER           → reply "not sure what this is, do you want me
                          to just save it as a task?"

    Schema additions (new tables):
      - direct_expenses (id, merchant, amount_cents, sales_tax_cents,
        category, expense_date, receipt_storage_path, created_by,
        team_member_id, created_at). Receipts bucket in Supabase Storage.
      - bill_captures (id, source: sms|whatsapp|dashboard, image_path,
        ocr_text, claude_classification jsonb, routed_to_type, routed_to_id,
        team_member_id, created_at). Full audit trail from photo →
        eventual action so you can always trace back what happened.

    Implementation plan:
      1. Dashboard drop-zone page (/dashboard/capture) — drag + drop or
         file input; uploads to Supabase Storage with a signed URL.
      2. POST /api/captures — accepts {image_path, source}, queues an
         Inngest event 'capture/classify'.
      3. Inngest function captureClassifyRunner:
           a. Load image from storage to a blob
           b. Send to Claude Sonnet vision API with a system prompt
              describing the classification taxonomy + asking for a
              structured JSON output (intent + fields)
           c. Based on intent, call the right service (createTask,
              createEvent, createDirectExpense, sendVendorIntakeLink,
              etc.)
           d. Write to bill_captures for audit
           e. Send confirmation SMS/WhatsApp back: "Got the receipt
              from <merchant> for $<amount>, logged as an expense."
      4. SMS/WhatsApp inbound: Twilio carries media attachments via
         MediaUrl0, MediaUrl1 params in the webhook. When present,
         download the image to Storage, then call the same capture
         pipeline. Reply with the confirmation when done.
      5. Update Claude passthrough system prompt so it knows the
         'capture' flow exists (so it can answer "did you file that
         screenshot I sent?" with list_captures or similar).

    Claude vision cost:
      - Each image is ~1500-2000 tokens of image data (small) + a few
        hundred tokens of prompt + output.
      - At Sonnet pricing: ~$0.01-0.02 per classification.
      - 20 receipts/day = $6-12/mo. Fits inside $20 cap comfortably.

    Sales tax / W9 decision logic (for "BILL" classification):
      - If receipt has sales tax amount → it's a PRODUCT sale from a
        registered retailer → direct_expense, no W9 needed.
      - If no sales tax AND service-like language ("consulting",
        "performance", "labor", "services rendered") → BILL: SERVICE,
        W9 needed, route to vendor intake.
      - Ambiguous → ask user in a confirmation SMS: "Looks like a bill
        from <merchant> for $<amount>. Is this a service (needs W9) or
        a product (no W9)?"

    Privacy / safety:
      - Images stored in a private Storage bucket.
      - Claude sees the image once per classification — not persisted
        in Anthropic's systems beyond the API call (per their data use
        policy).
      - PII in receipts (card last 4, address) stays in our DB — same
        treatment as existing vendor data.

    Total build estimate: ~6-8 hours. NOT a one-evening build; pair
    with coffee.

    Prerequisite Jason decisions:
      - Confirm the W9-vs-no-W9 heuristic matches his mental model.
        Verify with a CPA if scaling — this is our interpretation, not
        tax advice.
      - Pick the MMS strategy: Twilio Programmable Messaging supports
        MMS on US long codes + TF numbers (once TFV clears). WhatsApp
        supports media natively.
      - Decide whether to support email attachments too (via the
        deferred Resend inbound entry above — unified flow).
